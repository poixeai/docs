Poixe AI 是一层 **API 转发与调度层**，聚合并提供多家主流模型厂商的 API 能力。由于 Poixe 需要在不同厂商之间实现稳定的转发、计费与风控，我们在兼容性上会做一些 **工程取舍**（例如计费困难、安全或稳定性策略）。因此，部分能力在 Poixe 环境下可能会被**限制、禁用或表现不同**。

本章节主要说明两类内容：

* **Poixe 的兼容性取舍**：哪些功能在 Poixe 环境下受限/不可用，以及原因与替代建议。
* **已知的上游注意点**：一些来自厂商原生接口的“坑位提示”（并非 Poixe 引入），用于帮助你少踩坑、快速定位问题。由于上游实现与文档会变化，这些提示可能不全，仅供参考。

## 设计原则：最小侵入，不做协议转换

Poixe 的核心设计原则是：**最小侵入性（Minimal Invasiveness）**。

这意味着：

* 我们不会将一种协议的请求 **转换成另一种协议** 再调用上游。
  例如：不会把 `OpenAI Chat Completions` 的请求转换为 `OpenAI Responses` 协议再转发。
* Poixe 只做**转发、鉴权、调度、计费与日志**，上游调用使用的是**厂商原生提供的协议与行为**。

因此，当你选择某个协议进行请求时，你需要参考该协议在**具体厂商**上的官方文档实现细节：

* 如果你使用 `OpenAI Chat Completions` 协议调用 OpenAI 模型：请直接参考 OpenAI 官方文档。
* 如果你使用同样的协议调用 Azure OpenAI：请参考 Azure 提供的对应文档与参数约束。
* 如果你想通过 `OpenAI Chat Completions` 协议调用 Claude 模型：请参考 Anthropic 提供的对应接口说明（包括其兼容接口的特性与限制）。

<Callout type="info" icon="info-circle">
  Poixe 系统是直接对接各个厂商提供的协议接口，而不是由 Poixe 在中间做语义转换，所以协议差异需要以各厂商文档为准。
</Callout>
